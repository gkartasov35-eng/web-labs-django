# Информация для защиты проекта

## Что это за проект

Это учебный сайт на Django, в котором доделаны лабораторные работы №5, №6 и №7 по веб-программированию.

## Лабораторная работа №5

Требование: подключить базу данных, проверить регистрацию пользователя, проверить шифрование пароля, создать дополнительные модели и миграции.

Реализовано:

- база данных подключена через Django ORM;
- по умолчанию используется SQLite-файл `db.sqlite3`;
- регистрация реализована через `RegisterForm`, основанную на `UserCreationForm`;
- пользователь создаётся через стандартную Django-модель `User`;
- пароль не сохраняется открытым текстом, а хэшируется встроенным механизмом Django;
- добавлены модели `UserProfile`, `Feedback`, `Post`, `LikeDislike`;
- созданы и применяются миграции.

Что сказать на защите:

> В пятой лабораторной я подключил базу данных к Django-проекту. Регистрация пользователя проходит через форму `RegisterForm`. Django сам хэширует пароль, поэтому в базе хранится не исходный пароль, а хэш. Для проверки используется метод `check_password`. Также я добавил дополнительные модели и миграции.

## Лабораторная работа №6

Требование: создать форму или систему like/dislike, которая передаёт данные в JSON и обновляет страницу без перезагрузки.

Реализовано:

- форма обратной связи отправляется через `fetch`;
- like/dislike отправляются через AJAX;
- сервер возвращает `JsonResponse`;
- счётчики лайков и дизлайков обновляются без перезагрузки страницы;
- добавлен endpoint для получения актуальных реакций.

Что сказать на защите:

> В шестой лабораторной я реализовал обмен данными без перезагрузки страницы. На фронтенде используется `fetch`, а Django-view возвращает `JsonResponse`. После клика по like/dislike данные сохраняются в БД, а на странице обновляются только счётчики.

## Лабораторная работа №7

Требование: разбить проект на микросервисы, упаковать сервисы в Docker-контейнеры и реализовать локальное развёртывание.

Реализовано:

- основной сервис: Django-приложение;
- дополнительный микросервис: `notification_service` на Flask;
- сервисы запускаются через `docker-compose.yml`;
- для Django есть отдельный `Dockerfile`;
- для notification-service есть отдельный `Dockerfile`;
- добавлен GitHub Actions workflow для проверки тестов;
- добавлена подготовка к хостингу через `render.yaml`.

Что сказать на защите:

> В седьмой лабораторной проект разделён на два сервиса. Первый сервис — основное Django-приложение, второй сервис — микросервис уведомлений. Django при отправке формы может обратиться к микросервису по HTTP. Оба сервиса упакованы в Docker-контейнеры и запускаются одной командой через `docker compose up --build`.

## CI/CD

В проект добавлен отдельный workflow GitHub Actions:

```text
.github/workflows/ci-cd.yml
```

Что делает pipeline:

1. запускается при `push` и `pull_request`;
2. устанавливает зависимости проекта;
3. выполняет `python manage.py check`;
4. применяет миграции;
5. запускает тесты;
6. собирает Docker-образ;
7. после успешной проверки может вызвать Render Deploy Hook для автоматического деплоя.

Что сказать на защите:

> CI/CD реализован через GitHub Actions. Сначала выполняется CI: проверка Django-проекта, миграции, тесты и сборка Docker-образа. После успешной проверки на основной ветке может выполняться CD: GitHub Actions вызывает Render Deploy Hook, и хостинг автоматически пересобирает и публикует приложение.

## Команды для демонстрации

Локальный запуск без Docker:

```bash
cd config
pip install -r requirements.txt
python manage.py migrate
python manage.py runserver
```

Локальный запуск через Docker:

```bash
cd config
docker compose up --build
```

Проверка Django:

```bash
python manage.py check
python manage.py test
```
